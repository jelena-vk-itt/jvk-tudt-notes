  1. Introduction

    Introduction

    Like other types of software development, web development consists of a number of stages, some of which may overlap. Different sources may define or name the stages differently, however this is not important - the main purpose of naming the stages is to describe the process of development in terms of its constituent activities and the order in which they roughly take place. The stages are:

    • inception and planning
    • design
    • development
    • stabilisation
    • deployment
    • maintenance
    In this lecture we describe all the stages of development but with particular focus on the early ones, inception and planning and design.

  2. Inception and planning

    Inception and planning

    This is the lifecycle stage that deals with idea generation and the compilation of requirements. Initial requirements are developed and refined by taking various factors into consideration:

    • purpose - this needs to be clear and specific in type (entertainment, education, commercial, communication) and aim
    • initial requirements (customer's, own)
    • the audience (questionnaire, survey, interviews)
    • the competition (identify gaps and weaknesses and focus on own strengths and uniqueness)
    • presentation style
    • infrastructure integration
    • value
    Resulting from this stage is a set of tangible requirements of three types: content-related, functional and quality attributes (non-functional). For example:
    • the domain and scope of the content (content)
    • the functionality that will be provided (functional) - described in the form of use cases
    • the intended audience should be clearly addressed (quality attribute)
    • the website should be interesting and engaging for the audience (quality attribute)
    • usability of functions, e.g. intuitiveness of workflows (quality attribute)
    • usability of information e.g. easy to access and understand (quality attribute)
    • performance e.g. pages load quickly enough (quality attribute)

  3. Design

    Design

    A website has a number of aspects that must be designed at this stage, both in their own right and integrated into a seamlessly presented whole:

    • content
      • organise into folders
      • maintain an inventory (name, filename and type, size, location, author/owner, destination page)
    • structure
      • this needs to be intuitive for the user
      • a good way to organise the pages is to use a tree structure, reflected in the URL paths (see picture)
      • keep a one-topic-to-one-page mapping
      • strike a balance between depth and breadth (see picture)
    • page layout
      • designed in three stages:
        • wireframing – skeleton layout, used at the very early stage of design
        • user experience (UX) design (tools: Balsamiq, Mockingbird, Visio, pen and paper), concerned with
          • OS-native look-and-feel
          • device hardware (size, available buttons etc.)
          • mock-ups (not detailed design)
        • user interface (UI) design, concerned with
          • detailed design, including user interaction 'skeleton' (often templates are used)
          • professional look
          • consistency across website
      • some tips for user-friendly page content:
        • identify your website on every page
        • identify your website's purpose on the main pages
        • user- and search-engine-friendly page title
        • make pages that load quickly
        • ensure that the website is up to date
        • use conventional names for common pages (e.g. home, about, contact)
        • place objects in conventional places (logo, navbar, shopping cart, log-in button)
    • navigation
      • minimal: provide navigation to main pages e.g. homepage from every page
      • ideal: navigation structures that reflect the site structure
      • give clues as to 'where the user is' e.g. highlight the link to the current page (the user may not enter your site at the home page)
      • encourage exploration of the site with navigation bar and other links
      • for vast structured content, include a table of contents and ‘Up’ ‘Back’ ‘Next’ links
      • make links that are recognisable as links (underlined, different colour from other text, third colour if visited)
      • make navigation consistent and predictable
      • provide a site-map
      • provide a search facility
      • test with real users
    • media
    • accessibility
      • site should look good on various devices and window sizes
      • avoid frames
      • provide a printable version
      • do not interfere with browser functions:
        • pop-up or new window hyperlinks - tell user
        • avoid fixed-size fonts as user cannot zoom these
        • do not disable right-click, back or forward buttons

  4. Development

    Development

    This is the central part of the lifecycle. There are a number of stages that the product goes through before it is deployed:

    • prototype - proof of concept
    • alpha - core functionality implemented but bugs, including serious ones, still present
    • beta - all functionality implemented, initial testing performed and some bugs still present
    • release candidate - in the absence of new bugs appearing during final testing, this version will be released
    Prototype development may start in parallel with design.

  5. Stabilisation

    Stabilisation

    This part of the lifecycle

    • consists of testing and bug-fixing
    • should start in parallel with development, intensifying as the code matures
    • is the main activity from the alpha product stage and on

    Some test tools for web development:

    • Testflight (iOS)
    • launchpadapp.com (Android)
    • vessel.io (iOS, Android)
    • hockeyapp.com (iOS, Android, Windows)

    As with any software product, the website should be tested both from a content/functional point of view and with respect to quality attributes. Also, it needs to be tested at different levels:

    • unit
      • validation of HTML and CSS
      • testing of client-side scripts
      • testing of server-side scripts
      • proofing of content
    • system
      • testing across all platforms (e.g. browsers and devices) reasonably expected to be used
      • testing of navigation (including downloads)
      • accessibility testing (e.g. screen reading software, zoom, alt tags on images)
    • user
      • recruit testers representative of user base
      • create exhaustive test scenarios and a means of clear communication of results, including
        • content useful and accurate with respect to declared scope (or not)
        • declared functionality present in expected form (or not)
        • particular quality attributes (consumability of content, GUI aesthetics, ease of use, speed of loading) satisfactory (or not)
        • general impression of website on tester (would they return to it, recommend it to others etc.)
        • open-ended request for suggestions
      • let the testers perform some unstructured use of the website

  6. Deployment

    Deployment

    Deployment means making the website available to the intended audience.

  7. Maintenance

    Maintenance

    This consists of fixing any bugs that are found after deployment, as well as upgrades. Some considerations:

    • someone needs to be responsible for it and access should be controlled
    • make it easier by establishing procedures for backup, version control, rollback etc.
    • keep the website up to date by implementing a method for updating content (including tests and proofing)
